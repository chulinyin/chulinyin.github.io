导航守卫

刚开始看官方文档的时候我以为自己不会用到这个功能，没想到今天正好露姐说需要对后台项目进行更严格的权限控制，于是就正好用到了，这里有一些让人出乎意料坑。

原来开发的后台项目使用了单页系统，使用 vue router 的 hash 模式进行页面跳转控制，权限控制是简单地通过「页面入口是否显示在左侧菜单栏」控制的，而左侧菜单栏是通过访问后端接口获取数据后再渲染的。即，如果某个人没有某个页面的访问权限，后端就不会返回该页面的入口在菜单栏中的配置信息，菜单栏渲染后也不会显示该页面的访问入口。

但是这样会存在一个问题：当我手动输入无访问权限页面的地址时，依然可以跳转到该页面，也就是说此时压根就不存在权限控制。

所以为了实现权限控制，有两种可行的方式：

  1. 由后端进行权限控制，前端只需把 vue-router 实例化时的模式改为 history 模式即可。此时页面进行跳转时会先访问后端服务器，后端服务器返回前端单页入口文件之后，前端才会进行路由控制渲染。所以此时后端需要对访问路径进行权限检测，如果没有权限则返回 404 什么的。
  2. 由前端进行权限控制，此时便只能通过导航守卫（ Navigation Guards ）中的 router.beforeEach 方法进行权限控制了。

本来应该由后端进行权限控制比较合理一些，但是原来的后台项目的接口地址是直接按模块划分，且没有统一放在 /api 下。因此，如果由后端进行权限控制的话，可能会出现页面地址和接口地址重合的情况，于是就只能选择第二种方案了。下面是代码：

关于权限控制，突然又发现了一个坑。如果一个页面的路由地址存在于左侧菜单栏，而是由某个页面的按钮进行跳转的，即页面的入口不是左侧菜单栏，这时候应该如何进行权限控制呢？举例来说，假设左侧菜单栏有一个用户列表的页面，然后从用户列表页面可以跳转到用户详情页，即该详情页的入口只有用户列表页面。

此时，如果完全按照上述代码进行权限控制的话，跳转到详情页永远都是不成功的，因为详情页的地址永远不会存在于菜单栏对象中的。我现在使用的权宜之计是，对这种页面入口不在菜单栏的路由添加 meta: { requiresAuth: false } 字段，然后再在 router.beforeEach 的回调函数中获取 meta 进行额外的条件判断。

但是上面这种方法还是会存在最开始的问题：相当于没有对入口不在菜单栏的页面做权限控制。

2018.10.20 更新

昨天的更新启发了我，权限控制的正确判断逻辑应该是：如果某个页面不需要权限控制，那就直接跳转；如果页面的入口依赖于某个菜单栏中的页面，即也需要做权限控制时，应该判断依赖是否存在于菜单栏对象中，而不是页面本身的地址是否存在于菜单栏对象中；最后才是默认页面需要权限控制，且入口在菜单栏中，所以只需判断页面的地址是否存在于菜单栏对象中即可。所以，对于入口依赖于某个菜单栏中的页面，只要再在 meta 对象中增加所依赖入口的地址即可。
