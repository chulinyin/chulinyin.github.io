初见HTTP（2）

发表于 2016-05-28   |   分类于 HTTP   |     |   阅读次数

又是在图书馆渡过了平淡又充实的一天,基本上一直在一边看kindle的《图解HTTP》，一边做一些笔记，以便以后查阅起来方便点。单是看进度条的话，现在已经看了60%了，也就是说再看个两三天就能把HTTP最基础的部分掌握了。等看完这个就计划了解下AngularJS或者Vue或者Bootstrap，再然后就打算开始自己写组件了。

下面是今天的一些总结笔记：

HTTP协议确实很简单易用，但是也正因为这个，导致了一些它在安全方面的缺陷。所以当我们对安全性要求比较高的时候就不再单纯地只使用这个协议了，一般我们使用HTTPS。这两个不光看起来比较像，其实HTTPS就是HHTP协议再加上SSL（安全套接层）协议。
那么HTTP到底哪里不好，HTTPS又是哪里好呢？

1.1 HTTP通信使用明文（不加密），内容可能会被窃听。

因为按照TCP/IP协议族的工作机制，通信内容在所有的通信线路上都有可能被有心人看到。（用抓包或者嗅探器工具）

1.2 HTTPS使用加密处理，防止内容被破解

虽然加密后的报文本身还是会被有心人看到，但是那时候他看到的是一串混乱的字符串，如果不破解的话，肯定看不懂（相当于战争时候用的电报，需要有一张解码表才能知道电报信号的真正意思，不然你得到的只是无意义的滴滴声罢了）

2.1 HTTP协议对于任何人都可以发起请求

也就是说，服务器只要接收到请求报文，就会返回响应报文（当然是在发送端IP地址和端口号没有被服务器设定限制访问的情况下）可是这样就会造成下面的情况：

  • 可能是已伪装的服务器或客户端
  • 无法确定通信的对方是否有访问权限
  • 无法判定请求来自何方，出自谁手
  • 即使无意义的请求也照单全收（无法阻止海量请求下的DOS攻击）

2.2 HTTPS查明对方的证书

因为SSL提供了证书的手段（证书由值得信任的第三方机构颁发，用以证明服务器和客户端是实际存在的），只要确认通信方持有证书即可。

3.1 HTTP协议接收到的内容可能有误

用HTTP协议的话，即使请求或响应的内容遭到篡改，用户也无法知道。因为没有任何办法可以确认’发出的请求/响应’和’接收到的请求/响应’是前后相同的。
当然也可以用用MD5和SHA-1等散列值校验的算法以及用来确认文件的数字签名方法，但还是需要用户自己去检查验证下载的文件是否就是原来服务器上的文件。

3.2 HTTPS协议可以防止篡改

使用HTTPS（SSL提供认证和加密处理及摘要功能），可确保完整性。

不过，如果HTTPS这么好，那还有HTTP什么事？HTTPS应该也有很多不如HTTP的地方吧？明天再继续看下去吧~
