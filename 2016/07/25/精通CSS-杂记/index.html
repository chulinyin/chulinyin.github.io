id和class哪家强

一般原则是，类应该应用于概念上相似的元素，这些元素可以出现在同一页面上的多个位置，而ID则应该应用于不同的唯一元素。只有在绝对确定这个元素只会出现一次的情况下，才应该使用ID。如果你认为以后可能需要相似的元素，就使用类。

不要过度使用类，会在文档中添加无意义的代码。
不要过度使用div,这是代码结构不合理且过分复杂的一个信号。

样式规范

程序员、内容管理员和其他前端开发人员可能需要了解代码的元素和设计是如何工作的，而建立样式规范是维护或实现站点的好方法。可以确保网站开发以可控的方式进行，同时防止样式随着时间的推移变得散乱。还非常有助于向新职员或承包商介绍他们不熟悉、可能是很复杂的系统。

  1. 静态样式规范：一个文档、网页或小型站点
    解释代码和站点的视觉设计是如何组合在一起的。
    • 最基本的样式规范应该描述一般性设计原则，如如何处理标题和其他排版元素、网格结构的设计方式是什么，以及如何使用哪个调色板。
    • 好的样式规范还应描述文章、新闻条目和通知等重复元素的处理方法，决定他们应该如何实现和不应该如何实现。
    • 更详细的样式规范可以包含编码标准，使用的HTML/CSS版本、选择的可访问性级别、浏览器支持细节和一般的编码最佳实践。
  2. 动态样式规范
    因为及时更新静态样式规范需要花费大量精力，所以它们常常很快就与当前网站不同步了。所以使用动态的样式规范（“模式组合”）

模式组合是一个页面或一系列页面，它们使用当前的样式表显示站点上可能出现的每种样式排列组合，从标题级别和文本样式直到特定的内容和布局类型。

三、可视化格式模型

IE盒模型与标准盒模型

  • IE: ele = width + margin(width=content+padding+border)
  • W3C： ele = width+padding+border+margin(width=content)

解决方案：box-sizing:border-box;

外边距叠加

  • 一个元素在另一个元素上面，上一个元素的底外边距与下一个元素的顶外边距发生叠加。
  • 一个元素包含在另一个元素中（假设没有内边距或边框将外边距隔开），它们的顶/底外边距也会发生叠加。
  • 外边距甚至可以与本身发生叠加，当元素没有边框或内边距，但有外边距时，上下会叠加。如果叠加后的外边距碰到另一个元素的外边距，它们还会发生叠加

注意：只有*普通文档流中的块级元素框的垂直外边距会发生外边距叠加。行内框、浮动框或绝对定位框之间的外边距不会叠加。

八、布局

所有CSS布局技术的根本都是3个基本概念：定位、浮动和外边距操纵

！计划布局

要创建可伸缩且容易维护的CSS系统，首先应该检查设计，寻找重复的模式（包括页面结构中的模式或在站点中元素反复出现的方式），注意结构和意义而不应该太关注表现形式。

  1. 【建筑物外墙】划分大的结构性区域，如容器、页眉、内容区域、页脚。
  2. 【内部承重墙】在内容区域本身开始建立网络结构，寻找内容区域共同的特征而不是视觉表现。
  3. 【建筑物隔断墙】在各个内容区域中寻找不同的布局结构，将这些结构与前一步结合可以形成每个页面的结构计划。（可以开始设计结构和尺寸）
  4. 结构设计完后，可以开始关注不同类型的内容。给每一个内容块起一个有意义的名称，然后分析它们的关系。
  5. 查看每个内容块的结构，看看不同的类型中是否有共同的模式。尽量采用一般的类名，然后根据上下文应用样式。
  6. 找出模式并确定命名约定后，最好马上开始定义将使用的元素。

基于浮动的布局

最容易使用，也是最可靠的。为了解决浮动元素不再对包围它们的块框产生任何影响的问题，需要对布局中各个点上的浮动元素进行清理。但常见的做法不是连续地浮动和清理元素，而是浮动几乎所有东西，然后在整个文档的“战略点”（如页脚）上进行一次或两次清理。还可以使用溢出方法(overflow:hidden)清理某些元素的内容。

两列浮动布局

中间会有20px视觉隔离带是为了防止内容扩展导致浮动元素下降
设置display:inline是为了防止IE中的双外边距浮动产生的bug

三列浮动布局

可以使用与两列布局技术相同的CSS，将次要内容向左浮动，主要内容向右浮动。然后在主内容div中，将主div向左浮动，次div向右浮动。其实就是把主内容区域分成两列，从而形成三列效果。

流式（百分比）布局

尺寸用百分数设置，流式布局可以非常高效地使用空间。但是，在窗口宽度较小的时候，行变得非常窄，很难阅读。在多列布局中尤其如此。所以，有必要添加以像素或em为单位的min-width。
而如果设计横跨浏览器整个窗口，行就变得太长，也很难阅读。

  1. 不要横跨浏览器而是让容器只跨越宽度的一部分，如85%
  2. 还可以考虑用百分数设置内边距和外边距，可以防止列太快地变得过宽
  3. 对于非常严重情况，可以选择为容器设置最大宽度max-width。

弹性（em）布局

虽然流式布局可以充分利用可用空间，但是在大分辨率显示器上，行仍然会过长。相反，在窄窗口中或者在增加文本字号时，行会变得非常短，内容很零碎。

弹性布局相对于字号（而不是浏览器宽度）来设置宽度。以em为单位设置宽度，可以确保字号增加时整个布局随之扩大。这可以使长行保持在可阅读的范围。

但是弹性布局也不能充分利用可用空间，另外因为文本字号增加时整个布局会加大，所以弹性布局会变得比浏览器窗口宽，导致水平滚动条出现。为了防止这种情况，可能需要在容器div上添加100%的max-width。

技巧是设置基字号，让1em大致相当于10像素。比如大多数浏览器上的默认字号是16像素，10像素大约是16像素的62.5%，所以在主体上将字号设置为62.5%即可。

CSS框架与CSS系统

框架多有助于提高生产力的特性（全局reset、全站排版处理…），但是框架也会改变编写标记的方式，破坏表现与意义的分离。如果选用某个框架，你可能会在每个项目中都使用它，因此思维方式很可能会逐渐僵化。（“如果你有一把锤子，那么什么东西看起来都像是钉子。”）

而CSS系统是一个工具箱，其中包含可重用的样式和标记模式，可以用它们开发站点专用的框架。它能提供CSS框架的优点，同时避免其缺点。

九、bug和修复bug

hack确实是有力的工具，但是需要谨慎地应用，而且一般是作为最后的手段来使用。更为重要的技能是跟踪、隔离和识别bug。只有在知道bug是怎么回事后，才能想办法对付它。

最常见的CSS问题并非来源于浏览器bug，而是来源于对CSS规范的理解不完整。为了避免这些问题，在处理CSS bug时最好假设是自己做错了什么事。只有在确定不是自己的错之后，才应该考虑是否是浏览器bug的结果。

！常见的CSS问题

  1. 由代码中的打字和语法错误造成的。如：忘了分号。

    使用CSS3分析工具检查代码，然后查阅规范。

  2. 特殊性（优先级）和分类次序的问题

    property: value !important; > 内联 > id > class、伪类、属性选择器 > 标签选择器、伪元素选择器

  3. 垂直外边距叠加的问题？
    如果元素没有垂直边框或内边距，那么它的高度就是它包含的子元素的顶部和底部边框边缘之间的距离。因此，包含的子元素的顶部和底部外边距就突出到容器元素外边。

    一个简单的解决方案：添加一个垂直边框或内边距.parent{padding:1px;}，外边距就不在叠加了。

不要把浏览器测试留到项目快结束的时候，应该采用连续测试方法，在项目开发过程中用所有主流浏览器检查页面，这样就不会在项目快结束的时候又突然发现很多问题。

布局（IE6、7）

IE使用布局概念控制元素的尺寸和定位。那些“拥有布局”的元素负责本身及其子元素的尺寸设置和定位。如果一个元素“没有拥有布局”，那么它的尺寸和位置由最近的拥有布局的祖先元素控制。默认情况下拥有布局的元素：

  • body html(标准模式)
  • table tr td
  • img hr
  • input select texarea button
  • firame embed object applet
  • marquee

设置以下属性也会自动使元素拥有布局：

  • float: left/right
  • display: inline-block
  • width: any
  • height: any
  • zoom: any
  • writing-mode: tb-rl
  • overflow: hidden/scroll/auto
  • min-width: any
  • max-width: any except none

！布局造成的问题：

  1. 如果段落拥有布局，它就被限制为矩形，因此阻止文本围绕浮动元素
  2. 对于某些元素，布局像是清理了包含元素的浮动，就像设置了overflow:hidden。
  3. 在IE6中，拥有布局的元素会错误地扩展尺寸以便适应内容的尺寸吗，而不是溢到外面。
  4. 拥有布局的元素不会收缩
  5. 相对定位的元素没有布局
  6. 在没有布局的块级链接上，单击区域只覆盖文本
  7. 在滚动时，列表项上的背景图像间歇性的显示和消失。

hack和过滤器

利用CSS具有很强的向前兼容性（如果浏览器不理解某个选择器，则忽略整个规则；如果不理解某个属性或值，会忽略整个声明），添加新的选择器、属性和值应该不会对老式浏览器产生严重的影响。

因此使用不被支持的CSS作为过滤机制是一种相对安全的方法。但是使用依赖于解析bug的过滤器是一种有点儿冒险的方法，因为它们依赖于bug而不是特性。如果解析bug被修复了，而你试图修复的bug还没有得到解决，那么可能会再次遇到问题。

hack和过滤器常常会使你的代码失效，作为一般规则，应该使用依赖于不被支持的CSS的过滤器，而不是依赖于某种浏览器bug的过滤器，这可能比较安全。最好完全避免使用过滤器。

明智地使用hack和过滤器

不要过分信赖与hack和过滤器。如果只能应用某种hack或过滤器，那么需要以明智且受控制的方式应用它。（hack往往相当复杂，使代码难以阅读）

  • 如果CSS文件短小简单，并且只需要使用很少的hack，那么将这些hack放在主css文件中并添加注释说明可能是安全的。
  • 如果CSS文件很长很复杂，或者需要使用的hack比较多，那么最好将它们放在它们自己的样式表中。

藏猫猫bug（IE6）

现象：一个浮动元素后面跟着一些非浮动元素，然后是一个清理元素，所有这些元素包含在一个设置了背景颜色或图像的父元素中。如果清理元素碰到了浮动元素，那么中间的非浮动元素看起来消失了，隐藏到了父元素的背景颜色或图像后面，只有在重新加载页面时才再度出现，
