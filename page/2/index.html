线性表（List）

线性表：零个或多个数据元素的有限序列。（点名册、星座列表）

线性表的个数n(n>=0)定义为线性表的长度，当n=0时，称为空表。

有序，前驱元素，后继元素， 个数>=0，数据类型相同。

在较复杂的线性表中，一个数据元素可以由若干个数据项组成。

线性表的抽象数据类型

  • 创建和初始化( createList(), initList() )
  • 判断是否为空表( listEmpty() )
  • 重置为空表( clearList() )
  • 根据位序得到数据元素( getElem() )
  • 查找是否存在( locateElem() )
  • 线性表长度( listLength() )
  • 插入和删除( listInsert(), listDelete() )

算法综述

概念

算法：是解决特定问题的求解步骤的描述，在计算机中表现为指令的有限序列，并且每条指令表示一个或多个操作。（泡妞的技巧和方式，对于给定的问题，是以可以有多种算法解决的）

指令：能被人或机器等计算装置执行。

算法特征

  1. 输入：>=0
  2. 输出：>=1
  3. 有穷性：指算法在执行有限的步骤之后，自动结束而不会出现无限循环，并且每个步骤在可接受的时间内完成。（一定要可接收啊～）
  4. 确定性：每个步骤都具有确定的含义，无二义性；在一定条件下只有一条执行路径，相同的输入只能有唯一的输出结果；每个步骤都应该被精确定义而无歧义。
  5. 可行性： 每一步都必须是可行的，即每一步都能够通过执行有限次数完成。

算法设计要求

  1. 正确性：指算法至少应该具有五个基本特征
    • 无语法错误
    • 对合法输入能产生满足要求的输出
    • 对非法输入能够产生满足规格的说明
    • 对故意刁难的测试输入都有满足要求的输出结果
  2. 可读性：便于阅读、理解和交流
  3. 健壮性：当输入不合法时，算法也能做相关处理，而不是产生异常、崩溃等结果。
  4. 时间效率高和存储量低

数据结构基本概念和术语

数据结构（data struct）：（ 研究非数值计算的程序设计问题中的操作对象，以及他们之间的关系和操作等相关问题。）是相互之间存在一种或多种特定关系的数据元素的集合。

数据

数据（data）：描述客观事物的符号，是计算机中可操作的对象，能被计算机识别，并输入给计算处理的符号集合。（人、动物…..）(用户、商品、物流)

数据对象（data object）：是性质相同的数据元素的集合，是数据的子集。（学生群体、老师、员工……）（用户）

数据元素（data element）：是组成数据的、有一定意义的基本单位，在计算机中通常作为整体处理。也被称为记录。（学生A、学生B、学生C……）（用户A、用户B）

数据项（data item）：一个数据元素可以由若干个数据项组成。是数据不可分割的最小单位。（学号、姓名、身份证……）（账号密码）

结构

结构（struct）：指各个组成部分相互搭配和排列的方式。不同数据元素之间存在的特定关系。

逻辑结构：数据对象中数据元素之间的相互关系。

  1. 集合结构
  2. 线性结构（一对一）
  3. 树形结构（一对多）
  4. 图形结构（多对多）

物理结构：数据的逻辑结构在计算机中的存储形式

  1. 顺序存储结构（手拉手）：把数据元素存放在地址连续的存储单元里，其数据间的逻辑关系和物理关系是一致的。
  2. 链式存储结构（手拿个人信息[指针]）：把数据元素存放在任意的存储单元里，这组存储单元可以是连续／不连续的。

数据类型

数据类型：一组性质相同的值的集合，及定义在此集合上的一些操作总称。

类型：用于说明变量或表达式的取值范围和所能进行的操作。

抽象数据类型（ADT）：指一个数学模型及定义在该模型上的一组操作。（体现了问题分解、抽象和信息隐藏的特性）

抽象：抽取出事物具有的普遍性的本质。

《思维的乐趣》——积极的结论

发表于 2016-12-06   |   分类于 读书随感   |     |   阅读次数

开始第二遍看《思维的乐趣》了，我会对我觉得有意思的部分做摘录，然后讨论一番，希望能够帮助我在思维中享受到某些乐趣。

假设我们说话要守信义，办事情要有始有终，健全的理性实在是必不可少。

有了健全的理性之后，我们就不会轻易许下诺言，轻易开始做事情。因为你会先考虑那个诺言和那件事情是否合理，是否能够做到？

有关理性，哲学家有很多讨论，但根据我的切身体会，它的关键是：凡不可信的东西就不信。但这一点有时候不容易做到，因为会导致悲观和消极，从理性和乐观两样东西里选择理性颇不容易。理性就像贞操，失去了就不会再有；只要碰上了开心的事，乐观还会回来的。不过这一点很少有人注意到。从逻辑上说，从一个错误的前提什么都能推出来；从实际上看，一个扯谎的人什么都能编出来。所以假如你失去了理性，就会遇到大量令人诧异的新鲜事物，从此迷失在万花筒里，直到碰上了钉子。假如不是遇到了林彪事件，我至今还以为自己真能保卫毛主席哩。

关于责任的一点思考

发表于 2016-12-04   |   分类于 日记   |     |   阅读次数

一点琐事引发的思考：人和人的思考方式天差地别，但是在某些事情上的想法和行为却又出奇的一致。比如有的人倾向于积极地为自己行为所产生的结果负责，而有的人倾向于减轻甚至逃避这种责任。

今天中午，一位室友在晾衣服的时候，衣服了掉下来，不小心打碎了另一位室友的碗。我告诉他，碗碎了。然后他说，“没事，反正他也不用这个碗了。”

这是一件很小的事情，但是这里面所展现出的问题，我却在很多人身上见过。毫无疑问，我的这位室友在这件事情中是属于后一类人的。如果非要较真，他随口说出的那句话无疑是没有道理的，且让人听了不舒服。因为这里面存在这样一种逻辑：“我的行为使你的财产受损，但因为我认为那个财产是你目前不需要的，所以我不必为此负责，或者可以少负一点责任。”这里存在两个问题，第一，你怎么知道这个财产别人不需要了？第二，即使他确实用不到那个财产了，那难道就意味着你可以破坏而不赔偿？

按道理说，如果一个人使他人的财产受到损失，他的第一反应不应该是感到愧疚，表示歉意，然后主动弥补以获得他人的原谅吗？为什么人可以很自然甚至无意识地逃避这种责任呢？而且这似乎是一种普遍现象。

如果行为所产生的结果让自己产生了巨大痛苦，此时内心的自我逃避和免责似乎是可以理解的，因为这样可以防止承受不住压力的精神崩溃。但为什么明明是自己承受范围之内的事情，有些人的意识还是依然会不由自主地逃避呢？

以上想法是在较真情况下的思考。因为当事人没有在场，所以不能说我那室友随口所说的话是他真实的想法。

URL (stable)

方法

url.parse(urlString[, parseQueryString[, slashesDenoteHost]])
  • parseQueryString: 布尔值，默认为 false 。如果为true，则返回的 URL 对象的 query 属性奖总是被设为 querystring 模块的 parse() 方法返回的对象。如果为 false ， 则返回的 URL 对象的 query 属性将是一个未被解析的字符串。
  • slashesDenoteHost: 布尔值，默认为 false 。如果为 true ，则字符 // 之后， 下一个 / 之前的字符串将被解释为 host 。例如，给 //foo/bar ，返回结果将是 {host: 'foo', pathname: '/bar'} 而不是 {pathname: '//foo/bar'} 。

概念：

stream: 在计算机科学中，一个stream是一个随时间推移提供的数据元素的序列。一个 stream 可以被认为是一个传送带上一次处理一个而不是大批量的物品，在Unix以及相关基于 C 语言的系统中，一个 stream 是一个数据源或数据库，通常是单个字节或字符。 Streams 是当读取或写入文件，或在网络套接字上通信时通信的抽象使用。标准 streams是提供给所有程序的 3 位 stream 。（在计算机编程中，标准 streams 是，当一个电脑程序开始执行时，预连接程序和它的环境之间的输入输出通信通道。三个输出输入连接被称为：标准输入、标准输出、标准错误。）
